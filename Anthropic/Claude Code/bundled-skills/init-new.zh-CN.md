---
name: init
description: Initialize a new CLAUDE.md file with codebase documentation.
---
为此存储库设置一个最小的 CLAUDE.md（以及可选的技能和挂钩）。 CLAUDE.md 会加载到每个 Claude 代码会话中，因此它必须简洁 — 仅包含 Claude 如果没有它就会出现 get 错误的内容。

## 第 0 阶段：检查现有的 CLAUDE.md

在询问任何问题之前，请检查 CLAUDE.md 是否已存在于项目根目录中（仅 `cat ./CLAUDE.md` — 仅项目根文件有效；暂时不要探索树）。这分支到第一阶段。

## 第 1 阶段：询问要设置什么

使用 AskUserQuestion 找出用户想要什么。您提出哪个问题取决于阶段 0。使用**仅 Q1** 调用 AskUserQuestion — 不要在同一调用中包含 Q2。仅在看到问题 1 的答案后才询问问题 2，因为“让克劳德决定”会跳过它。

在回答第一个问题之前，请将此入门知识打印为普通的助理文本，以便初次用户了解这些术语：

> 快速上下文：
> - **CLAUDE.md** 文件为 Claude 提供有关项目、个人工作流程或组织的持久指示。克劳德在每次课程开始时都会阅读它们。
> - **技能**是克劳德在任务匹配时自动调用的打包指令，或者您使用斜线命令触发的指令（例如 `/frontend-design`、`/commit-push-pr`）。
> - **Hooks** 允许您在生命周期事件上自动运行 shell 命令：当 Claude 无法输入时会收到 get 通知、编辑后自动格式化、提交前强制检查 - 这些都是确定性的，Claude 无法跳过它们。

**如果 CLAUDE.md 已经存在**，询问：
- “我找到了现有的 CLAUDE.md。你想做什么？”
  选项：“审查并改进它”| “放下它，设置其他的东西”| “重新开始（替换它）”
  改进描述：“探索代码库中的更改并提出对现有文件的有针对性的编辑。”
  left it 的描述：“跳过 CLAUDE.md。直接看技能和钩子。”
  重新开始的描述：“丢弃它并写入新文件。”
  路由：
  -“回顾并改进”→跳过Q1/Q2；探索（第 2 阶段），提出单个第 3 阶段精简版问题，然后进入第 4 阶段的 diff-proposal，然后进入第 8 阶段。
  - “保留”→ 跳过 Q1，询问 Q2（将其第四个选项重命名为“都不 — 跳过设置”）。如果他们选择“都不 — 跳过设置”，请直接跳到第 8 阶段：“无需设置 — 您的 CLAUDE.md 未更改。”否则：阶段 2 → 阶段 3 提案（无间隙填充访谈）→ 每个队列的阶段 6/7 → 阶段 8。对于阶段 7 的挂钩目标文件默认值，请将此路径视为 "project" (`.claude/settings.json`)。
  - “重新开始”→ 继续下面的 Q1，就好像不存在文件一样。

**如果 CLAUDE.md 不存在**（或用户选择“重新开始”），则询问：
- 问题一：“/init 应该设置哪些 CLAUDE.md 文件？”
  选项：“项目 CLAUDE.md”| “个人 CLAUDE.local.md”| “既项目+个人”| “让克劳德决定”
  项目描述：“检查到源代码控制的团队共享指令——架构、编码标准、通用工作流程。”
  个人描述：“您对此项目的私人偏好（gitignored，不共享）——您的角色、沙箱 URL、首选测试数据、工作流程怪癖。”
  让 Claude 决定的描述：“最快的路径 - 项目 CLAUDE.md 加上适合此存储库的任何技能或挂钩。没有后续问题；您将在编写之前批准所有内容。”
  如果用户选择“让 Claude 决定”，则跳过 Q2 — 将其视为没有技能/钩子限制的项目 CLAUDE.md。

- Q2：“还设置了技能和挂钩？”
  选项：“技能+挂钩”| “仅技能” | “仅挂钩” | “都不是，只是克劳德.md”
  技能描述：“当任务匹配时，或者用斜杠命令触发时，克劳德会自动调用打包指令（例如 `/frontend-design`、`/commit-push-pr`）。”
  挂钩描述：“在工具事件上运行的确定性 shell 命令（例如，每次编辑后的格式）。Claude 无法跳过它们。”
  Q2 是一个提示，而不是一个过滤器——第 3 阶段提出了适合代码库的内容并记录任何偏差。

## 第 2 阶段：探索代码库

启动子代理来调查代码库，并要求它读取关键文件以了解项目：清单文件（package.json、Cargo.toml、pyproject.toml、go.mod、pom.xml 等）、README、Makefile/build 配置、CI 配置、现有 CLAUDE.md、 .claude/rules/、AGENTS.md、.cursor/rules 或 .cursorrules、.github/copilot-instructions.md、.windsurfrules、.clinerules、.mcp.json。

检测：
- 构建、测试和 lint 命令（尤其是非标准命令）
- 语言、框架和包管理器
- 项目结构（带有工作区的 monorepo、多模块或单个项目）
- 与语言默认值不同的代码样式规则
- 不明显的陷阱、所需的环境变量或工作流程怪癖
- 现有的 .claude/skills/ 和 .claude/rules/ 目录
- 格式化程序配置（prettier、biome、ruff、black、gofmt、rustfmt 或统一格式脚本，如 `npm run format` / `make fmt`）
- Git 工作树用法：运行 `git worktree list` 检查此存储库是否有多个工作树（仅在用户想要个人 CLAUDE.local.md 时相关）

请注意您无法仅从代码中弄清楚的内容 - 这些将成为面试问题。

## 第三阶段：填补空白

使用 AskUserQuestion 收集编写优秀 CLAUDE.md 文件所需的知识和技能。只问事情代码无法回答。

如果用户选择项目 CLAUDE.md，两者都选择，或者“让 Claude 决定”：询问代码库实践 - 非显而易见的命令、陷阱、分支/PR 约定、所需的环境设置、测试怪癖。跳过自述文件中已有的内容或清单文件中显而易见的内容。不要将任何选项标记为 "recommended" - 这是关于他们的团队如何工作，而不是最佳实践。

如果用户选择个人 CLAUDE.local.md 或两者：询问它们，而不是代码库。不要将任何选项标记为 "recommended" - 这是他们的个人偏好，而不是最佳实践。问题示例：
  - 他们在团队中的角色是什么？ （例如“后端工程师”、“数据科学家”、“新员工入职”）
  - 他们对该代码库及其语言/框架的熟悉程度如何？ （这样克劳德就可以校准解释深度）
  - 他们是否有克劳德应该知道的个人沙箱 URL、测试帐户、API 关键路径或本地设置详细信息？
  - 仅当第 2 阶段发现多个 git 工作树时：询问它们的工作树是否嵌套在主存储库（例如 `.claude/worktrees/<name>/`）或兄弟/外部（例如 `../myrepo-feature/`）内。如果嵌套，向上的文件遍历会自动找到主存储库的 CLAUDE.local.md — 无需特殊处理。如果是同级/外部，个人内容应位于主目录文件中（例如 `~/.claude/<project-name>-instructions.md`），并且每个工作树都会获得一个导入它的单行 CLAUDE.local.md 存根：`@~/.claude/<project-name>-instructions.md`。切勿在项目 CLAUDE.md 中导入 put — 这会将个人引用检查到团队共享文件中。
  - 有什么沟通偏好吗？ （例如，“简洁”、“始终解释权衡”、“不要在最后总结”）

如果用户在第 0 阶段选择“审查和改进”：只问一个问题 —“自从编写此 CLAUDE.md 以来，团队的工作方式是否发生了任何变化（新约定、命令、陷阱）？”带有选项“不，没有任何改变”| “是的——让我描述一下”。如果他们选择“是”，请在继续之前询问发生了什么变化（自由文本）。然后跳到第 4 阶段。

**根据第 2 阶段的发现和填补空白的答案综合提出建议。** 对于每个项目，选择适合证据的工件类型：

  - **Hook** — 确定性、快速、每次编辑的 shell 命令（格式化、检查已更改的文件）。
  - **技能** — 按需多步骤工作流程（`/verify`、`/deploy-staging`、会话报告）。
  - **CLAUDE.md note** — 塑造行为但不强制执行的指导（惯例、沟通方式）。

包含 Q1 隐含的 CLAUDE.md 文件（项目、个人、两者或“让 Claude 决定”→ 项目）作为提案的第一个项目符号，以及每个项目符号所涵盖内容的一行摘要。然后列出技能/挂钩/注释。在“Leave it”路径上，省略 CLAUDE.md 文件项目符号和注释（第 4 阶段不会运行）。在 Q1 = 仅限个人的“全新开始”路径上，添加一个项目符号，指出现有项目 CLAUDE.md 将保持不变（他们选择不将其替换为项目文件）。

提出适合的建议。如果用户给出了 Q2 提示，而您的提案偏离了它（例如，他们说“仅钩子”，但不存在任何钩形），请在提案顶部的一行中如此说，并无论如何提出更合适的工件。

**将提案打印为普通助理文本**，每项一个项目符号：

> 这是我的设置：
> • **[工件类型：文件/挂钩/技能/注释]** — [单行描述]
> • …

然后调用 AskUserQuestion，提出一个简单的问题（“这看起来正确吗？”）和“看起来不错 - 继续”等选项 | “放下钩子”| “放下技能”。不要使用 `preview` 字段 — 该提案已在回滚中可见。该工具会自动添加 "Other" 选项以进行自定义调整。

**根据已接受的提案构建偏好队列**。每个条目：{类型：钩子|技能|注释、描述、目标文件、任何来自第 2 阶段的详细信息，例如实际的测试/格式化命令}。第 6 阶段和第 7 阶段的 hooks 子项目符号消耗此队列；直接在已批准提案的文件项目符号上进行第 4/5 阶段；无论队列内容如何，​​第 7 阶段的 GitHub-CLI 和 linting 检查都会运行。

## 第 4 阶段：编写 CLAUDE.md（如果批准的提案包含它，或者在“审查和改进”路径上）

在项目根目录编写一个最小的 CLAUDE.md。每行都必须通过这个测试：“删除这个会导致克劳德犯错误吗？”如果没有，就剪掉。

如果用户在第 0 阶段选择“审查并改进它”：不要写新内容 - 阅读现有文件，与第 2 阶段的发现和第 3 阶段精简版答案进行比较，并提出具体的添加/删除作为差异，并为每个添加/删除添加一行原因。现有文件为基线；你的工作是找出缺失的、过时的或臃肿的东西。打印差异后，在编写任何内容之前调用 AskUserQuestion（“应用这些编辑？”，其中包含“应用全部”|“让我选择哪个”|“跳过 - 保持原样”等选项）。

**使用目标为 CLAUDE.md 的第 3 阶段首选项队列中的 `note` 条目**（团队级别注释） - 将每个条目作为简洁的行添加到最相关的部分中。这些是用户希望 Claude 遵循但不需要保证的行为（例如，“在实施之前提出计划”、“解释重构时的权衡”）。留下个人针对性的笔记对于第 5 阶段。

包括：
- 克劳德无法猜测的构建/测试/lint 命令（非标准脚本、标志或序列）
- 与语言默认值不同的代码样式规则（例如，“优先选择类型而不是接口”）
- 测试说明和怪癖（例如，“使用以下命令运行单个测试：pytest -k 'test_name'”）
- 回购礼仪（分支命名、PR 约定、提交风格）
- 所需的环境变量或设置步骤
- 不明显的问题或架构决策
- 现有 AI 编码工具配置的重要部分（如果存在）（AGENTS.md、.cursor/rules、.cursorrules、.github/copilot-instructions.md、.windsurfrules、.clinerules）

排除：
-逐个文件的结构或组件列表（克劳德可以通过阅读代码库发现这些）
- 克劳德已经知道的标准语言约定
- 通用建议（“编写干净的代码”，“处理错误”）
- 详细的 API 文档或长引用 - 使用 `@path/to/import` 语法（例如 `@docs/api-reference.md`）来按需内联内容而不膨胀 CLAUDE.md
- 经常更改的信息 — 使用 `@path/to/import` 引用源，以便 Claude 始终读取当前版本
- 长教程或演练（移至单独的文件并在技能中引用 `@path/to/import` 或 put）
- 清单文件中显而易见的命令（例如，标准“npm test”、“cargo test”、"pytest"）

具体来说：“在 TypeScript 中使用 2 个空格缩进”比“正确格式化代码”更好。

不要重复自己，也不要编造诸如“常见开发任务”或“开发提示”之类的部分 - 仅包含在您阅读的文件中明确找到的信息。

文件前缀为：```
# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.
```对于具有多个问题的项目，建议将说明组织到 `.claude/rules/` 作为单独的重点文件（例如，`code-style.md`、`testing.md`、`security.md`）。它们会与 CLAUDE.md 一起自动加载，并且可以使用 `paths` frontmatter 将范围限定到特定文件路径。

对于具有不同子目录的项目（monorepos、多模块项目等）：提到可以添加子目录 CLAUDE.md 文件以用于特定于模块的指令（当 Claude 在这些目录中工作时，它们会自动加载）。如果用户愿意，可以主动创建它们。

## 第5阶段：编写CLAUDE.local.md（如果批准的提案包含它）

在项目根目录写入一个最小的 CLAUDE.local.md。该文件会与 CLAUDE.md 一起自动加载。创建后，将 `CLAUDE.local.md` 添加到项目的 .gitignore 中，以使其保持私有状态。

**使用目标为 CLAUDE.local.md 的第 3 阶段首选项队列中的 `note` 条目**（个人级别注释） - 将每个条目添加为简洁的行。如果用户在第一阶段选择仅限个人，则这是注释条目的唯一消费者。

包括：
- 用户的角色和对代码库的熟悉程度（以便 Claude 可以校准解释）
- 个人沙箱 URL、测试帐户或本地设置详细信息
- 个人工作流程或沟通偏好

保持简短——只包含能让 Claude 的响应对该用户明显更好的内容。

如果第 2 阶段发现多个 git 工作树并且用户确认他们使用同级/外部工作树（未嵌套在主存储库内）：向上文件遍历将不会从所有工作树中找到单个 CLAUDE.local.md。将实际的个人内容写入 `~/.claude/<project-name>-instructions.md` 并使 CLAUDE.local.md 成为导入它的单行存根：`@~/.claude/<project-name>-instructions.md`。用户可以将此单行存根复制到每个同级工作树。从来没有在项目 CLAUDE.md 中导入 put 这个。如果工作树嵌套在主存储库中（例如，`.claude/worktrees/`），则不需要特殊处理 - 主存储库的 CLAUDE.local.md 会自动找到。

如果 CLAUDE.local.md 已经存在：阅读它，提出具体的添加，并且不要默默地覆盖。

## 第 6 阶段：建议和创造技能（如果批准的提案包含任何技能）

技能增加了克劳德可以按需使用的能力，而不会在每次会话中都显得臃肿。

**首先，使用第 3 阶段首选项队列中的 `skill` 条目。** 每个排队的技能首选项都会成为根据用户描述定制的 SKILL.md。对于每个：
- 根据首选项命名（例如，"verify-deep"、"session-report"、"deploy-sandbox"）
- 写正文使用采访中用户自己的话以及第二阶段发现的任何内容（测试命令、报告格式、部署目标）。如果偏好映射到现有的捆绑技能（例如，`/verify`），请编写一项项目技能，在顶部添加用户的特定约束 - 告诉用户捆绑的技能仍然存在，并且他们的技能是附加的。
- 如果首选项未指定，请快速跟进（例如，“应验证深度运行哪个测试命令？”）

**然后，当您发现以下情况时，建议队列之外的其他技能**：
- 特定任务的参考知识（子系统的约定、模式、风格指南）
- 用户希望直接触发的可重复工作流程（部署、修复问题、发布流程、验证更改）

对于每项建议的技能，请提供：名称、一行目的以及为什么它适合此存储库。

如果 `.claude/skills/` 已存在技能，请先查看它们。不要覆盖现有技能——只提出新的技能来补充已有的技能。

在 `.claude/skills/<skill-name>/SKILL.md` 创建每个技能：```yaml
---
name: <skill-name>
description: <what the skill does and when to use it>
---

<Instructions for Claude>
```默认情况下，用户（`/<skill-name>`）和克劳德都可以调用技能。对于具有副作用的工作流程（例如 `/deploy`、`/fix-issue 123`），请添加 `disable-model-invocation: true`，以便只有用户可以触发它，并使用 `$ARGUMENTS` 接受输入。

## 第 7 阶段：提出额外优化建议

告诉用户，既然 CLAUDE.md 和技能（如果选择）已经到位，您将建议一些额外的优化。

检查环境并询问您发现的每个差距（使用 AskUserQuestion）：

- **GitHub CLI**：运行 `which gh` （或 Windows 上的 `where gh`）。如果它丢失并且项目使用 GitHub（检查 github.com 的 `git remote -v`），请询问用户是否要安装它。解释一下 GitHub CLI 让 Claude 可以直接帮助提交、拉取请求、问题和代码审查。

- **Linting**：如果第 2 阶段没有发现 lint 配置（项目语言没有 .eslintrc、ruff.toml、.golangci.yml 等），请询问用户是否希望 Claude 为此代码库设置 lint。解释 linting 可以及早发现问题，并为 Claude 提供对其自己编辑的快速反馈。

- **提案来源的挂钩**（如果已批准的提案包含任何挂钩）：使用第 3 阶段首选项队列中的 `hook` 条目。如果第 2 阶段找到格式化程序并且队列没有格式化挂钩，请提供编辑时格式化作为后备。

  对于每个钩子首选项（来自队列或格式化程序回退）：

  1.目标文件：默认基于Phase 1 CLAUDE.md选择——项目→`.claude/settings.json`（团队共享，已提交）；个人 → `.claude/settings.local.json`。仅询问用户是否在第一阶段选择了 "both" 或者偏好不明确。对所有钩子询问一次，而不是针对每个钩子。

  2. 从首选项中选择事件和匹配器：
     -“每次编辑后”→ `PostToolUse` 与匹配器 `Write|Edit`
     -“当克劳德完成时”/“在我回顾之前”→ `Stop` 事件（在每个回合结束时触发 - 包括只读回合）
     - “运行 bash 之前” → `PreToolUse` 与匹配器 `Bash`
     - “提交之前”（字面量 git-commit 门）→ **不是钩子。json 钩子。** 匹配器无法通过命令内容过滤 Bash，因此无法仅针对 `git commit`。将其路由到 git 预提交钩子（`.git/hooks/pre-commit`，husky，预提交框架）——提议编写一个。如果用户实际上的意思是“在我审查并提交 Claude 的输出之前”，那就是 `Stop` — 探测以消除歧义。
     探究偏好是否不明确。

  3. **加载钩子引用**（每次 `/init` 运行一次，在第一个钩子之前）：使用 `skill: 'update-config'` 和以以下参数开头的参数调用 Skill 工具`[hooks-only]` 后跟您正在构建的内容的一行摘要 - 例如，`[hooks-only] Constructing a PostToolUse/Write|Edit format hook for .claude/settings.json using ruff`。这会将钩子架构和验证流程加载到上下文中。后续的钩子会重用它——不要重新调用。

  4. 按照技能的 **“构建 Hook”** 流程：重复数据检查 → 构建此项目 → 管道测试原始 → 包装 → 写入 JSON → `jq -e` 验证 → 实时验证（对于可触发匹配器上的 `Pre|PostToolUse`）→ 清理 → 移交。目标文件和事件/匹配器来自上面的步骤 1-2。

  在继续之前，先对每个 "yes" 进行操作。
