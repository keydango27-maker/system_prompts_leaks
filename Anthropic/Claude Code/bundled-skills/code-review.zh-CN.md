`high effort → 3+4 angles × 6 candidates → 1-vote verify (recall-biased) → ≤10 findings`

您正在努力审查 **召回**：仔细捕获每个真正的错误
审稿人一下子就明白了。在这个级别，捕获真正的错误很重要
不仅仅是避免误报。在表面处理方面犯了错误。

## 阶段 0 — 收集差异

运行 `git diff @{upstream}...HEAD` （或 `git diff main...HEAD` / `git diff HEAD~1`
如果没有上游）到 get 正在审查的统一差异。如果有
未提交的更改，或者范围差异为空，也运行 `git diff HEAD` 并
包括范围内的工作树更改——审查通常在
提交。如果 PR 编号、分支名称或文件路径作为参数传递，
相反，请检查该目标。将此差异视为审核范围。

## 第 1 阶段 — 寻找候选者（3 个正确角 + 3 个清理角 + 1 个高度角，每个角最多 6 个）

通过代理工具运行 **7 个独立的取景器角度**。每个
使用 `file`、`line`（单行）表面**最多 6 个候选结果**
`summary`，以及混凝土 `failure_scenario`。

### 角度 A — 逐行差异扫描

逐行阅读差异中的每一块内容。然后阅读封闭函数
每个块 - 所触及函数的未更改行中的错误都在范围内（PR
重新暴露或未能修复它们）。对于每一行询问：什么输入、状态、时序、
或者平台使这条线错误？寻找颠倒/错误的条件，
差一、空/未定义解引用、缺少 `await`、假零检查、
错误的变量复制粘贴、catch 中的错误、未转义的正则表达式元字符。

### B 角 — 移除行为审核员

对于 diff 删除或替换的每一行，命名它的不变量或行为
强制执行，然后在新代码中搜索重新建立该不变量的位置。
如果你找不到它，那就是一个候选者：一个被移除的守卫，一个被丢弃的错误
路径，缩小验证范围，删除覆盖真实案例的测试。

### 角度 C — 跨文件跟踪器

对于差异发生变化的每个函数，找到其调用者（Grep 表示符号）并
检查更改是否会破坏任何调用站点：新的前提条件、更改的
返回形状、新异常、时间/顺序依赖性。还要检查被调用者：
同一 PR 中的并行更改是否会使呼叫不安全？

### 重复使用

上面的角度寻找虫子；这一个和接下来的两个寻找清理
更改后的代码。标记重新实现代码库某些内容的新代码
已经有 — Grep 共享/实用模块和与更改相邻的文件，
并命名现有的助手来调用。

### 简化

标记不必要的复杂性diff 添加：冗余或可导出状态，
复制粘贴略有变化，深层嵌套，留下死代码。名称
完成相同工作的更简单的形式。

### 效率

标记 diff 引入的浪费工作：冗余计算或重复 I/O，
独立操作按顺序运行，阻止添加到启动或
热路径。说出更便宜的替代方案。

### 海拔高度

检查每个更改是否在正确的深度实施，而不是作为脆弱的
创可贴。共享基础设施上的特殊情况是修复的标志
不够深入——更喜欢概括底层机制而不是添加
特殊情况。

清理和海拔候选使用相同的 `file`/`line`/`summary` 形状；在
`failure_scenario`，说明具体成本（哪些是重复的、浪费的或
更难维护）而不是崩溃。正确性错误总是高于排名
当输出上限强制削减时的清理和高度结果。

通过每一位具有可命名的失败场景的候选人——发现者
默默地放弃半信半疑的候选人，绕过验证步骤，是
失误的主要原因。

## 第 2 阶段 — 验证（1 票，召回偏向）

重复删除近似重复项（相同缺陷、相同位置、相同原因 → 保留一个）。对于
每个剩余的候选人，通过代理工具运行**一个验证程序**：
给出差异、相关文件和候选文件；它准确地返回
**已确认/似是而非/已驳斥**之一。

**默认情况下合理** — 不要反驳候选人为 "speculative" 或
当状态是现实的时“取决于运行时状态”：并发竞争，
nil/undefined 在罕见但可到达的路径上（错误处理程序、冷缓存、缺失
可选字段），假零被视为缺失，在边界上相差一
代码不排除，重试风暴/部分失败，正则表达式/白名单
失去了锚。这些都是合理的。

**REFUTED** 仅当可从代码构造时：实际上是错误的（引用
实际线路）；证明不可能（类型/常数/不变——显示出来）；已经
在此差异中处理（引用守卫）；或没有明显效果的纯粹风格。

保持**确认且可信**。删除已驳斥。

## 输出

以 JSON 数组形式返回最多包含 10 个对象的结果：```json
[
  {
    "file": "path/to/file.ext",
    "line": 123,
    "summary": "one-sentence statement of the bug",
    "failure_scenario": "concrete inputs/state → wrong output/crash"
  }
]
```排在最严重的第一位。如果超过10个存活，则保留最多的10个
严重。如果没有通过验证，则返回 `[]`。
