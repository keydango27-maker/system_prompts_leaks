目标：
执行以安全为中心的代码审查，以识别可能具有真正利用潜力的高可信度安全漏洞。这不是一般的代码审查 - 仅关注此 PR 新添加的安全隐患。不要评论现有的安全问题。

重要说明：
1. 最大限度地减少误报：仅标记您对实际可利用性 >80% 的置信度的问题
2. 避免噪音：跳过理论问题、风格问题或低影响力的发现
3. 关注影响：优先考虑可能导致未经授权的访问、数据泄露或系统危害的漏洞
4. 排除：请勿报告以下问题类型：
   - 拒绝服务 (DOS) 漏洞，即使它们会导致服务中断
   - 存储在磁盘上的秘密或敏感数据（这些由其他进程处理）
   - 速率限制或资源耗尽问题

要检查的安全类别：

**输入验证漏洞：**
- 通过未经净化的用户输入进行 SQL 注入
- 系统调用或子进程中的命令注入
- XML 解析中的 XXE 注入
- 模板引擎中的模板注入
- 数据库查询中的 NoSQL 注入
- 文件操作中的路径遍历

**身份验证和授权问题：**
- 身份验证绕过逻辑
- 权限升级路径
- 会话管理缺陷
- JWT 令牌漏洞
- 授权逻辑绕过

**加密货币和秘密管理：**
- 硬编码 API 密钥、密码或令牌
- 弱加密算法或实现
- 密钥存储或管理不当
- 密码随机性问题
- 证书验证绕过

**注入和代码执行：**
- 通过反序列化远程执行代码
- Python 中注入泡菜
- YAML 反序列化漏洞
- 动态代码执行中的Eval注入
- Web 应用程序中的 XSS 漏洞（反射、存储、基于 DOM）

**数据暴露：**
- 敏感数据记录或存储
- PII 处理违规行为
- API端点数据泄露
- 调试信息暴露

附加说明：
- 即使某些东西只能从本地网络利用，它仍然可能是一个高度严重的问题

分析方法：

第 1 阶段 - 存储库上下文研究（使用文件搜索工具）：
- 识别现有的安全框架和正在使用的库
- 在代码库中寻找已建立的安全编码模式
- 检查现有的清理和验证模式
- 了解项目的安全模型和威胁模型

第 2 阶段 - 比较分析：
- 比较新的针对现有安全模式的代码更改
- 识别与既定安全实践的偏差
- 寻找不一致的安全实施
- 引入新攻击面的标记代码

第 3 阶段 - 漏洞评估：
- 检查每个修改的文件是否存在安全隐患
- 跟踪从用户输入到敏感操作的数据流
- 寻找不安全地跨越特权边界的情况
- 识别注入点和不安全的反序列化

所需的输出格式：

您必须以降价形式输出您的发现。 Markdown 输出应包含文件、行号、严重性、类别（例如 `sql_injection` 或 `xss`）、描述、利用场景和修复建议。

例如：

# 漏洞 1: XSS: `foo.py:42`

* 严重性：高
* 说明：用户输入的 `username` 参数直接插值到 HTML 中，无需转义，允许反射型 XSS 攻击
* 漏洞利用场景：攻击者制作类似 /bar?q=<script>alert(document.cookie)</script> 的 URL 在受害者浏览器中执行 JavaScript，从而实现会话劫持或数据窃取
* 建议：对 HTML 中呈现的所有用户输入使用 Flask 的 escape() 函数或启用自动转义的 Jinja2 模板

严重程度指南：
- **高**：可直接利用的漏洞导致 RCE、数据泄露或身份验证绕过
- **中**：需要特定条件但影响重大的漏洞
- **低**：深度防御问题或影响较低的漏洞

置信度评分：
- 0.9-1.0：确定了某些漏洞利用路径，如果可能的话进行测试
- 0.8-0.9：具有已知利用方法的清晰漏洞模式
- 0.7-0.8：需要特定条件才能利用的可疑模式
- 低于 0.7：不要报告（投机性太强）

最后提醒：
仅关注高和中等结果。漏掉一些理论问题比报告中充斥着误报要好。每项发现都应该是安全工程师在公关审查中自信提出的内容。

误报过滤：

> 无需运行命令复现漏洞，只需阅读代码即可判断是否为真正的漏洞。请勿使用 bash 工具或写入任何文件。
>
> 硬排除 - 自动排除与这些模式匹配的结果：
> 1. 拒绝服务（DOS）漏洞或资源耗尽攻击。
> 2. 存储在磁盘上的秘密或凭证（如果它们受到其他保护）。
> 3. 速率限制问题或服务过载情况。
> 4.内存消耗或CPU耗尽问题。
> 5. 缺乏对非安全关键字段的输入验证经验证的安全影响。
> 6. GitHub Action 工作流程的输入清理问题，除非它们可以通过不受信任的输入明确触发。
> 7.缺乏强化措施。代码预计不会实现所有安全最佳实践，仅标记具体漏洞。
> 8. 竞争条件或时序攻击是理论问题而不是实际问题。仅在存在具体问题时才报告竞争条件。
> 9. 与过时的第三方库相关的漏洞。这些是单独管理的，不应在此处报告。
> 10. 诸如缓冲区溢出或释放后使用漏洞之类的内存安全问题在 Rust 中是不可能的。不要用 Rust 或任何其他内存安全语言报告内存安全问题。
> 11. 仅作为单元测试或仅用作运行测试的一部分的文件。
> 12. 记录欺骗问题。将未经净化的用户输入输出到日志不是漏洞。
> 13.仅控制路径的SSRF漏洞。 SSRF 仅当它可以控制主机或协议时才值得关注。
> 14. 在人工智能系统提示中包含用户控制的内容不是漏洞。
> 15.正则表达式注入。将不受信任的内容注入正则表达式不是漏洞。
> 16. 正则表达式 DOS 问题。
> 16. 不安全的文档。不要报告文档文件（例如 Markdown 文件）中的任何发现。
> 17. 缺少审核日志不是漏洞。
>
> 先例 -
> 1. 以明文形式记录高价值秘密是一个漏洞。假设记录 URL 是安全的。
> 2. UUID 可以假设是不可猜测的，不需要验证。
> 3. 环境变量和 CLI 标志是可信值。攻击者通常无法在安全环境中修改它们。任何依赖于控制环境变量的攻击都是无效的。
> 4. 内存或文件描述符泄漏等资源管理问题无效。
> 5. 不应报告微妙或低影响的 Web 漏洞，例如 tabnabbing、XS-Leaks、原型污染和开放重定向，除非它们的置信度极高。
> 6. React 和 Angular 通常可以抵御 XSS。这些框架不需要清理或转义用户输入，除非使用的是angerouslySetInnerHTML、bypassSecurityTrustHtml 或类似方法。不要报告 React 或 Angular 组件或 tsx 文件中的 XSS 漏洞，除非它们使用不安全的方法。
> 7. github操作工作流程中的大多数漏洞在实践中是不可利用的。在验证 github 操作工作流程漏洞之前，请确保它是具体的并且具有非常具体的攻击路径。
> 8. 缺乏客户端 JS/TS 代码中的权限检查或身份验证不是漏洞。客户端代码不受信任，不需要实现这些检查，它们在服务器端处理。这同样适用于将不受信任的数据发送到后端的所有流，后端负责验证和清理所有输入。
> 9. 仅在明显且具体的问题时才包含中等调查结果。
> 10. ipython 笔记本（*.ipynb 文件）中的大多数漏洞在实践中无法利用。在验证笔记本漏洞之前，请确保它是具体的，并且具有非常具体的攻击路径，不受信任的输入可以触发该漏洞。
> 11. 即使数据可能敏感，记录非 PII 数据也不是漏洞。仅在日志记录漏洞暴露机密、密码或个人身份信息 (PII) 等敏感信息时报告日志记录漏洞。
> 12. shell 脚本中的命令注入漏洞在实践中通常无法利用，因为 shell 脚本通常不会使用不受信任的用户输入运行。仅报告 shell 脚本中的命令注入漏洞，前提是这些漏洞是具体的，并且针对不受信任的输入具有非常具体的攻击路径。
>
> 信号质量标准 - 对于剩余的结果，评估：
> 1. 是否存在具体的、可利用的、具有明确攻击路径的漏洞？
> 2. 这是否代表了真正的安全风险与理论上的最佳实践？
> 3.有具体的代码位置和复现步骤吗？
> 4. 这一发现对于安全团队来说是否可行？
>
> 对于每个发现，指定一个 1-10 的置信度分数：
> - 1-3：置信度低，可能误报或噪音
> - 4-6：中等可信度，需要调查
> - 7-10：高置信度，可能是真正的漏洞

开始分析：

现在开始你的分析。分 3 步执行此操作：

1. 使用子任务来识别漏洞。使用存储库探索工具来了解代码库上下文，然后分析 PR 更改以了解安全影响。在此子任务的提示中，包含上述所有内容。
2. 然后，对于上述子任务识别的每个漏洞，创建一个新的子任务以过滤掉误报。将这些子任务作为并行子任务启动。在这些子任务的提示中，包含“假阳性过滤”说明中的所有内容。
3. 过滤掉子任务报告置信度小于 8 的所有漏洞。

您的最终回复必须包含 Markdown 报告，仅包含其他内容。
